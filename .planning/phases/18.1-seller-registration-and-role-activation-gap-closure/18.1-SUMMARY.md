# Phase 18.1 Summary

Phase 18.1 closed the seller-registration gap by preserving the existing profile-settings entry path, adding a protected `/seller/register` route, and proving that seller-role activation is server-authoritative.

## Surviving Implementation Trail

1. `ProfileEditPage.tsx` keeps the profile-settings entry path for authenticated non-sellers. The `Register as Seller` button pushes `/seller/register` from profile settings instead of introducing a separate onboarding hub.
2. `src/pages/seller/register.tsx` wraps the seller registration screen in `useRequireAuth()` and only renders when `canRenderProtected` is true, so `/seller/register` is an authenticated route.
3. `SellerRegistrationPage.tsx` reuses the current registration form, checks for an existing seller profile on load, and recovers duplicate registration by re-fetching the seller profile after a `409` conflict so the user can continue profile setup instead of getting stuck.
4. On submit, the frontend first uses `usersApi.updateMe()` to sync display-name and slug changes on the user record, then calls `profileApis.createSellerProfile()` to create the seller profile. These are related UX steps, but they are not a single cross-tier atomic transaction.
5. The backend `POST /identity/seller-profiles` create route is guarded by JWT in the API gateway and forwards the authenticated user context to identity-service.
6. `CreateSellerProfile.command.handler.ts` executes the seller-profile creation inside `transactionService.execute(...)` and adds `UserRole.SELLER` in the same backend transaction when the user does not already have the role.

## Policy and State Guarantees

- Seller-role activation is server-authoritative. The browser never grants `UserRole.SELLER` directly.
- The transactional boundary is limited to seller-profile creation plus seller-role addition in identity-service. The separate frontend `usersApi.updateMe()` call must not be described as part of that backend transaction.
- Seller verification remains off by default: `isVerified: false`, `verifiedAt: null`.
- Payment onboarding remains incomplete by default: `stripeOnboardingComplete: false` and `paypalOnboardingComplete: false`.
- The seller registration screen explicitly tells users that verification and payment onboarding remain separate policy gates.
