# Phase 35 - UI Design Contract

> Visual and interaction contract for the profile-owner moodboard multi-upload composer and profile moodboard rendering polish. Generated by gsd-ui-phase and verified before planning.

---

## Scope

Phase 35 applies to:

| Surface | Files |
|---------|-------|
| Profile owner studio entry and moodboard modal replacement | `FE/artium-web/src/@domains/profile/views/ProfilePage.tsx` |
| Moodboard multi-upload composer | `FE/artium-web/src/@domains/profile/components/MoodboardDeviceUploadComposer.tsx` |
| Moodboard upload state hook | `FE/artium-web/src/@domains/profile/hooks/useProfileMoodboardUpload.ts` |
| Existing moodboard cards and profile overview/list rendering | `FE/artium-web/src/@domains/profile/components/MoodboardsSection.tsx` |
| Moodboard detail rendering | `FE/artium-web/src/@domains/profile/views/ProfileMoodboardDetailPage.tsx` |
| Profile API mapping/types for uploaded moodboard media | `FE/artium-web/src/@domains/profile/utils/profileApiMapper.ts`, `FE/artium-web/src/@domains/profile/types/index.ts`, `FE/artium-web/src/@shared/apis/profileApis.ts` |
| Upload transport and validation contract | `FE/artium-web/src/@shared/apis/profileMediaUploadApi.ts` |

This phase replaces the current pasted-cover-URL moodboard modal with a device-upload composer. It supports 1 to 10 uploaded media items, removal, ordering, cover selection, title, description, privacy, collaboration, and tags. It also polishes profile overview, moodboard list, and detail rendering so uploaded moodboard media appears from backend data instead of frontend-only placeholders.

Out of scope:
- Arbitrary pasted cover image URLs.
- Nested moodboard references.
- A new visual system outside the existing profile and Orders-inspired language.
- Heavy media editing, cropping, transcoding, or advanced gallery editing.

## Design System

| Property | Value |
|----------|-------|
| Tool | Tailwind + existing shared shadcn-compatible primitives |
| Preset | `profile-orders-moodboard-composer` |
| Component library | Radix primitives through `@shared/components/ui`; use existing `Dialog`, `Input`, `Textarea`, `Switch`, `Button`, and `Progress` before adding primitives |
| Icon library | `lucide-react` only |
| Font | Existing app sans font; do not introduce a new typeface |

Design sources:
- Phase 33 locks the backend upload proof model: moodboard upload accepts up to 10 files and creation consumes backend-issued `mediaIds` plus `coverMediaId`.
- Phase 34 locks the profile media composer language: calm white modal, slate surfaces, clear upload/progress/failure states, and no pasted URL fields.
- Orders workspace is the style reference for operational UI: white panels, slate text, compact status chips, inline errors, restrained progress indicators, and stable disabled states.

## Spacing Scale

Declared values:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, drag handle gaps, compact media badges |
| sm | 8px | Thumbnail grid gaps, inline status spacing, chip padding |
| md | 16px | Default form gaps, media row spacing, field spacing |
| lg | 24px | Panel padding, modal section gaps |
| xl | 32px | Desktop modal column gap and major internal rhythm |
| 2xl | 48px | Large viewport modal breathing room |
| 3xl | 64px | Page-level separation only; avoid inside composer |

Exceptions:
- Interactive controls must keep at least a 40px by 40px hit area.
- Media thumbnails may use stable square dimensions such as 72px, 96px, or 112px to prevent layout jumps.
- Detail-page media grids may use responsive CSS grid tracks, but gaps still stay on the spacing scale.

## Screen And Interaction Contract

### Entry And Modal Shell

- The owner-only studio tools area keeps the `New moodboard` entry point.
- The profile moodboards tab/page may also expose the same composer if the existing page has an owner creation affordance.
- Use the existing `Dialog` pattern and keep modal width close to `4xl`/`5xl` depending on content density.
- Desktop modal uses a two-zone layout:
  - Primary zone: upload queue, preview grid/list, cover/order controls, upload status.
  - Secondary zone: title, description, tags, privacy, collaboration, and footer actions.
- Mobile stacks the upload queue above metadata and keeps footer buttons readable without horizontal scrolling.
- Closing the dialog resets local composer state: selected files, upload progress, media IDs, cover selection, ordering, errors, and metadata values.

### Upload Queue

- Empty state is a multi-file dropzone with visible type/count/size copy and a `Choose files` button.
- Accept 1 to 10 files. Adding files above the limit must show an inline validation error and preserve already accepted media.
- Upload should start immediately after file selection/drop, mirroring Phase 34.
- Each queued item shows:
  - thumbnail/preview or media-type placeholder
  - original filename, file size, and media type
  - upload status
  - progress when uploading
  - retry/remove actions when failed
  - cover selection affordance when uploaded
  - reorder affordance when more than one uploaded item exists
- Progress may be aggregate plus per-item status. If the API only exposes aggregate progress for the batch, present an aggregate progress bar and per-item "waiting/uploaded/failed" labels after response/error handling.
- Metadata fields remain editable while upload is running.

### Cover Selection

- Default cover is the first successfully uploaded item.
- Users can select any successfully uploaded moodboard media item as cover.
- Cover must always be one of the submitted `mediaIds`; never a pasted URL or unrelated file.
- Selected cover has a visible `Cover` chip and accent ring.
- If the selected cover is removed, the next available uploaded item becomes cover automatically.
- If no uploaded item remains, the composer returns to a no-cover/no-publish state.

### Ordering And Removal

- Ordered media IDs submitted to the backend must match the visual order shown in the composer.
- Drag-and-drop reorder is preferred when straightforward. If drag-and-drop adds risk, up/down icon buttons are acceptable and must have accessible labels.
- Reordering controls are disabled while the relevant item is uploading if moving it would make state ambiguous.
- Removing an item preserves title, description, tags, privacy, collaboration, and other uploaded items.
- Removing failed items should be allowed without blocking other successfully uploaded files.

### Metadata Form

Fields in this order:
1. Title
2. Description
3. Tags
4. Private board
5. Allow collaborators

Rules:
- Title is required before submit.
- At least one uploaded media item is required before submit.
- Description and tags are optional.
- Tags remain one comma-separated text input; no chip editor in this phase.
- `Private board` and `Allow collaborators` use simple `Switch` rows aligned with existing shared controls.
- Do not include cover image URL or manual media URL fields.

### Footer Actions

- Footer actions live at the bottom of the metadata zone.
- `Cancel` is secondary and always visible.
- `Create moodboard` is the only primary CTA.
- Primary CTA is disabled when:
  - no title is present
  - no media item has uploaded successfully
  - any required upload is still in progress
  - a failed upload remains selected for submission
  - no cover can be resolved
  - submission is in progress
- While submitting, primary CTA label changes to `Creating...`.

## Moodboard Rendering Contract

### Profile Overview And Moodboard List

- `MoodboardsSection` should render uploaded media from backend moodboard `media` data when available.
- Existing stacked-card language should be preserved, but cover/secondary images should derive from uploaded moodboard media before falling back to `coverImageUrl` or placeholders.
- Private boards keep the compact `Private` chip.
- Cards should not show broken images if a media thumbnail is missing; use `url`/`secureUrl` fallback and then a quiet placeholder.
- Multiple uploaded media items should be hinted with stacked images and a concise count, not a separate gallery inside each card.

### Moodboard Detail Page

- The detail page should render the moodboard's uploaded media in display order.
- The selected cover should be visually prominent near the title or first grid item.
- Image and video items should use stable aspect ratios and avoid layout shift.
- Video items may render thumbnail/poster first, with playback controls only where the existing page layout can support them cleanly.
- Empty or unavailable media state should show a quiet inline message, not a full-page error.
- Existing artwork-card fallback content should not masquerade as uploaded moodboard media. If artworks remain present, clearly separate them from uploaded media.

## State Contract

| State | Trigger | UI Contract |
|-------|---------|-------------|
| Empty | Modal opened, no files chosen | Multi-file dropzone visible, metadata editable, create disabled |
| Validating | Files selected before upload begins | Brief inline pending state; preserve accepted queue |
| Uploading | `uploadProfileMoodboardMedia` request in flight | Aggregate progress visible, per-file rows stable, metadata editable |
| Partial failure | Some files failed validation/upload | Inline error, failed items marked, successful items preserved |
| Uploaded | One or more files uploaded and media IDs stored | Queue shows `Media ready`, cover selectable, create eligible once title exists |
| Reordering | User changes item order | Visual order updates immediately and submitted `mediaIds` follow it |
| Cover selected | User selects a media item as cover | `Cover` chip and accent ring move to selected item |
| Removing | User removes item | Remaining order and cover resolve deterministically |
| Submitting | `profileApis.createMoodboard` request in flight | Disable conflicting actions, keep layout stable, show `Creating...` |

Rules:
- Client validation may mirror `profileMediaUploadApi.ts`, but backend upload validation remains authoritative.
- The composer must submit `mediaIds` and `coverMediaId`, not `coverImageUrl`.
- Unsupported mixed-media combinations must fail with clear validation instead of silently dropping media.
- If backend returns ordered `media` rows, frontend rendering should trust backend order.

## Responsive Contract

| Surface | Contract |
|---------|----------|
| Desktop modal | Two zones with upload queue wider than metadata; no nested card stacks inside cards |
| Tablet | Queue and metadata may collapse when constrained; thumbnails stay stable |
| Mobile modal | Single column, upload queue first, metadata second, footer buttons wrap cleanly |
| Upload queue | Long filenames truncate; actions wrap without overlapping status/progress |
| Moodboard cards | Existing horizontal/large card layouts remain stable with uploaded media fallbacks |
| Detail page | Responsive grid preserves display order and avoids clipping videos/images |

Responsive rules:
- No horizontal scrolling inside the composer.
- Thumbnail grids/lists must define stable dimensions so progress labels, cover chips, and remove buttons do not resize the layout.
- Button text must remain fully readable on narrow screens.

---

## Accessibility Contract

- The modal needs a clear accessible title: `New moodboard`.
- Dropzone/file picker must be keyboard reachable even when drag/drop is supported.
- Reorder controls need accessible labels such as `Move media up` / `Move media down` if icon buttons are used.
- Icon-only remove/cover controls require screen-reader labels and visible hover/focus affordances.
- Upload progress and failures must not rely on color alone; pair color with labels.
- Disabled create state should be explained by adjacent status/helper text where possible.
- Focus should return to the triggering `New moodboard` control when the modal closes when practical.

---

## Data Truth Contract

- Device upload must use `uploadProfileMoodboardMedia`.
- Moodboard creation must use `profileApis.createMoodboard` with `mediaIds` and `coverMediaId`.
- The UI must not reconstruct a pasted cover URL or pasted media URL path.
- Created moodboards should continue using `mapMoodboardToProfileMoodboard`, expanded as needed to include backend `media` rows for overview/list/detail rendering.
- `ProfileMoodboard`, `MoodboardApiItem`, and moodboard detail rendering should expose enough ordered media metadata to render uploaded content accurately.
- Frontend-only gallery state must not be treated as persisted moodboard truth.
