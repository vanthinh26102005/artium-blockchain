## Database strategy and schema behavior

- `BE/libs/common/src/database/database.helper.ts` is the canonical `DB_STRATEGY` switch:
  - `DB_STRATEGY=SHARED` uses `SHARED_DB_HOST`, `SHARED_DB_PORT`, `SHARED_DB_NAME`, and per-service schemas created before TypeORM connects.
  - `DB_STRATEGY=ISOLATED` falls back to per-service `DB_HOST`, `DB_PORT`, and `DB_NAME` style wiring.
- `BE/docker-compose.shared.yml` models the shared-database intent by making each service `depends_on: db-shared`.
- `BE/docker-compose.isolated.yml` models isolated database availability, but only `identity-service` explicitly overrides `DB_STRATEGY: ISOLATED`; the other service `.env.compose` files still default to `DB_STRATEGY=SHARED`.
- Result: the repo contains both shared and isolated database topologies, but the isolated overlay is not a clean, universally switched runtime contract yet.

## Mounted files and runtime assumptions

| Runtime assumption | Current implementation | Evidence |
|---|---|---|
| GCS service account file must exist locally | `artwork-service` and `messaging-service` both mount `./config/gcs-service-account.json:/app/config/gcs-service-account.json:ro` and set `GCS_KEY_FILE=/app/config/gcs-service-account.json` | `BE/docker-compose.yml`, `BE/apps/artwork-service/.env.compose`, `BE/apps/messaging-service/.env.compose` |
| Most services run as live source mounts in dev | Compose mounts `.:/app` plus per-service `/app/node_modules` volumes | `BE/docker-compose.yml` |
| Compose startup ordering is part of current runtime behavior | `depends_on` ties services to `rabbitmq`, `redis`, `mailhog`, `db-shared`, or per-service DB containers depending on overlay | `BE/docker-compose.yml`, `BE/docker-compose.shared.yml`, `BE/docker-compose.isolated.yml` |
| Shared DB mode requires schema creation before ORM bootstrap | `ensureSchemaExists(...)` creates per-service schemas when `DB_STRATEGY=SHARED` | `BE/libs/common/src/database/database.helper.ts` |
