## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| config files -> dependency inventory | The artifact must reflect real dependency wiring rather than inferred or preferred architecture. |
| secret-bearing env vars -> published markdown | Only variable names and dependency roles should be documented; secrets themselves must stay out of artifacts. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-21-04 | Information Disclosure | external integration matrix | mitigate | Capture variable names, mounted file paths, and system roles only; never paste credential values or file contents. |
| T-21-05 | Tampering | dependency map | mitigate | Tie each dependency entry to source evidence from Compose, `.env.compose`, or code helpers. |
| T-21-06 | Spoofing | external dependency classification | mitigate | Separate internal-local-dev infra from third-party integrations explicitly so later phases do not confuse Mailhog or local Postgres with production providers. |
</threat_model>
