# Phase 34 - UI Design Contract

> Visual and interaction contract for the profile-owner moment device upload composer. Generated by gsd-ui-phase and verified before planning.

---

## Scope

Phase 34 applies to:

| Surface | Files |
|---------|-------|
| Profile owner workspace entry | `FE/artium-web/src/@domains/profile/views/ProfilePage.tsx` |
| Existing moments list after successful publish | `FE/artium-web/src/@domains/profile/components/MomentsSection.tsx` |
| Upload transport and validation contract | `FE/artium-web/src/@shared/apis/profileMediaUploadApi.ts` |
| Moment creation request contract | `FE/artium-web/src/@shared/apis/profileApis.ts` |

This phase replaces the current pasted-URL moment modal with a single-file device upload composer. It does not add moodboard creation, multi-file moment upload, cover selection, social editing tools, or public-facing marketing surfaces.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Tailwind + existing shared shadcn-compatible primitives |
| Preset | `profile-orders-composer` |
| Component library | Radix primitives through `@shared/components/ui`; use existing `Dialog`, `Input`, `Switch`, and shared button styling before adding new primitives |
| Icon library | `lucide-react` only |
| Font | Existing app sans font; do not introduce a new typeface for this composer |

Design sources:
- Phase 34 context decisions D-01 through D-16 are locked.
- Profile currently uses local modal state inside `ProfilePage.tsx`; the implementation may extract focused components, but the visual language must remain consistent with the existing profile workspace.
- Orders is the style reference for calm operational UI: white panels, slate text, restrained status chips, explicit loading and error states, and no dramatic gradient hero treatment.

## Spacing Scale

Declared values:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon and status-dot gaps, dense metadata separators |
| sm | 8px | Compact inline spacing, helper copy spacing, chip padding |
| md | 16px | Default form gaps, row spacing, field spacing |
| lg | 24px | Panel padding, section gaps inside modal |
| xl | 32px | Desktop modal column gap, major vertical rhythm |
| 2xl | 48px | Modal top-to-bottom breathing room on larger viewports |
| 3xl | 64px | Reserved for page-level separation only; not inside the composer |

Exceptions:
- Interactive controls must keep a minimum 40px by 40px hit area.
- Dropzone preview media may use fixed aspect ratios; internal padding still stays on the scale above.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Small metadata | 12px | 500 | 1.35 |
| Label | 12px | 600 | 1.2 |
| Section heading | 16px | 600 | 1.35 |
| Modal title | 24px | 600 | 1.2 |
| Numeric progress emphasis | 14px | 600 | 1.2 |

Rules:
- Use only normal in-app type sizes; no hero-scale headline inside the composer.
- Keep letter spacing at `0` except existing uppercase eyebrow labels, which may use light positive tracking already present in the app.
- Long filenames, captions, and location text must wrap or truncate without pushing buttons or progress bars out of alignment.
- Do not scale typography with viewport width.

## Screen And Interaction Contract

### Entry And Modal Shell

- The owner-only studio tools area keeps the `New moment` entry point.
- Use the existing `Dialog` pattern.
- Desktop modal target: large centered dialog near current `4xl` width, but visually converted from dramatic split treatment to a calm two-zone workspace.
- Mobile: same dialog content stacks vertically with the upload/status zone first and metadata second.
- Closing the dialog resets local composer state, including selected file, upload progress, `mediaId`, preview URL, and metadata values.

### Two-Zone Layout

| Zone | Role | Contract |
|------|------|----------|
| Primary zone | Upload preview and upload state | Shows dropzone, preview, file summary, status chip, progress bar, and retry/replace actions |
| Secondary zone | Metadata form | Holds caption, location, hashtags, pin toggle, and footer actions |

Layout rules:
- Desktop uses two columns with the upload/status panel slightly wider than the metadata panel.
- Mobile uses a single column stack with consistent white-panel rhythm.
- The two zones should read as sibling panels inside one modal, not as a left hero and right form split.
- Do not place cards inside cards; each zone is one panel.

### Empty Upload State

- Primary zone starts as a single-file dropzone.
- Support click-to-pick via hidden file input and drag/drop if native events are straightforward.
- Dropzone must communicate exact-one-file behavior.
- Accepted type/limit copy is visible inside the zone, not hidden behind tooltip-only help.
- Use an upload-related lucide icon if one exists; do not draw custom SVG.

### Uploading State

- File upload starts immediately after a valid file is selected or dropped.
- Publish is disabled while upload is in progress.
- Metadata fields remain editable during upload.
- Show local preview immediately when possible:
  - image: object URL preview
  - video: object URL preview with controls optional for pending state, but poster/thumbnail handling should not block the upload flow
- Show a linear progress bar plus numeric percent if progress events are available.
- Status chip style should follow Orders discipline: compact, quiet, readable, and not oversized.

### Uploaded State

- On success, store and visually confirm the uploaded file summary and `Upload complete` status.
- Publish becomes eligible when upload is complete and submission is not in progress.
- Replace action remains visible and clears the prior `mediaId` before re-uploading the new file.
- Preview remains primary; file metadata such as name, type label, and size may appear in a compact summary row.

### Failure And Validation States

- Client-side validation failures for type, size, or video duration render inline in the primary zone.
- Metadata values already entered must remain untouched after validation or upload failure.
- Failed state keeps the preview when practical and shows `Retry upload` and `Replace file`.
- Blocking error treatment uses a bordered inline panel or alert row with destructive color, not a browser alert and not modal-wide takeover.
- Retry should retry the same file when it still exists in local state.

### Metadata Form

- Fields in this order:
  1. Caption
  2. Location
  3. Hashtags
  4. Pin to profile
- Caption uses a multiline text area.
- Location and hashtags can share a two-column row on larger screens and stack on mobile.
- Hashtags remain one comma-separated text input; no chip editor in this phase.
- `Pin to profile` should stay a simple label plus `Switch` row aligned with existing shared controls.
- Do not include media URL, thumbnail URL, or manual media type toggle fields.
- Do not require caption, location, or hashtags before publish.

### Footer Actions

- Footer actions live at the bottom of the metadata zone.
- `Cancel` is secondary and always visible.
- `Publish moment` is the only primary CTA.
- Publish disabled states must cover:
  - no file selected
  - upload in progress
  - upload failed
  - replacement upload in progress
  - submission in progress
- While submitting, primary CTA label changes to `Publishing...`.

## State Contract

| State | Trigger | UI Contract |
|-------|---------|-------------|
| Empty | Modal opened, no file chosen yet | Dropzone visible, metadata editable but publish disabled |
| Validating | File selected before upload call begins | Brief inline pending state; do not flash a separate screen |
| Uploading | `uploadProfileMomentMedia` request in flight | Preview visible, progress bar visible, publish disabled, metadata editable |
| Uploaded | Upload succeeded and `mediaId` stored | Success chip, summary row, publish enabled if not submitting |
| Validation error | Client rejects type/size/duration | Inline destructive message, keep metadata, show replace action |
| Upload failed | Network/server upload error | Inline failure panel, keep metadata, offer retry and replace, publish disabled |
| Replacing | User chose new file after previous success/failure | Clear old `mediaId`, show new uploading/replacing state, keep metadata |
| Submitting | `profileApis.createMoment` request in flight | Disable all footer actions that would conflict, keep layout stable, show `Publishing...` |

Rules:
- Closing the modal exits all states and resets local form state.
- Only one file exists in local composer state at a time.
- Video duration may be collected when available, but the UI should not force manual duration entry unless implementation cannot derive it cleanly. If a manual field remains temporarily for implementation reasons, it should be treated as a fallback detail, not a primary control.

## Responsive Contract

| Surface | Contract |
|---------|----------|
| Desktop modal | Two columns, clear panel separation, no overflow collisions, footer actions stay easy to reach |
| Tablet | Two columns may collapse when space is tight; no clipped preview or crushed fields |
| Mobile modal | Single column stack, upload panel first, footer buttons wrap cleanly without overlapping |
| Dropzone | Copy and controls remain readable without crowding at narrow widths |
| Preview media | Maintain stable aspect ratio and avoid layout jumps when switching between image and video |

Responsive rules:
- Long filenames must truncate within the upload summary row instead of pushing action buttons off-screen.
- Buttons may wrap to a second line on mobile, but text must remain fully readable.
- No horizontal scrolling inside the composer.

---

## Accessibility Contract

- The modal needs a clear accessible title: `New moment`.
- Dropzone/file-picker interaction must remain keyboard reachable even if drag/drop is added.
- Icon-only affordances require visible tooltip or screen-reader text.
- Upload progress and failure states must not rely on color alone; pair color with label text.
- Disabled publish state should be understandable from adjacent status or helper copy, not only opacity.
- Focus should return to the trigger button when the modal closes when practical.

---

## Data Truth Contract

- Device upload must use `uploadProfileMomentMedia`.
- Moment creation must use the returned `mediaId` through `profileApis.createMoment`.
- The UI must not reconstruct a pasted URL flow as fallback behavior.
- Client-side file validation may mirror `profileMediaUploadApi.ts`, but backend upload validation remains authoritative.
- Successful publish should continue using existing profile moment mapping so the moments list updates without a full page redesign.
