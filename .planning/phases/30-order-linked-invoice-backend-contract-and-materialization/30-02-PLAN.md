<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> API gateway | Users can request any order ID; gateway must hide unauthorized orders before invoice RPC. |
| API gateway -> orders-service | Existing order authorization depends on orders-service returning order items with seller IDs. |
| API gateway -> payments-service | Payments-service materialization receives source order data only after authorization. |
| Frontend API type -> future UI | Phase 31 UI will trust this API shape, so it must match backend fields. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-30-05 | Information Disclosure | `GET /orders/:id/invoice` | mitigate | Call `getAuthorizedOrder(id, req.user?.id)` before payments RPC and preserve 404 behavior for unauthorized users. |
| T-30-06 | Tampering | source order payload | mitigate | Gateway builds the source DTO from the authorized backend order object, never from request body fields. |
| T-30-07 | Elevation of Privilege | frontend API | mitigate | Frontend method only calls the authenticated gateway endpoint; no direct payments-service invoice endpoint is added. |
</threat_model>
