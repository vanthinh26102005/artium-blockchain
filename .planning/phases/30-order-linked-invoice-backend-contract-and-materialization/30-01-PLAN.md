<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| API gateway -> payments-service | Payments-service receives an order snapshot only after gateway order authorization; it must not expose this RPC as a public authorization decision. |
| Payments-service -> database | Invoice creation must be idempotent so repeated reads do not duplicate financial records. |
| Order/payment data -> invoice DTO | The DTO becomes the financial source for Phase 31 UI; it must not invent client-only totals. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-30-01 | Spoofing | payments RPC | mitigate | Accept only sanitized source order data from gateway; no public payments HTTP route for arbitrary order invoice materialization in this phase. |
| T-30-02 | Information Disclosure | order invoice DTO | mitigate | DTO includes only invoice preview/export fields and payment identifiers already visible on order detail. |
| T-30-03 | Tampering | invoice materialization | mitigate | Derive totals and line items from source order data, use deterministic unique invoice number, and re-read on duplicate create. |
| T-30-04 | Repudiation | payment link | mitigate | Persist `paymentTransactionId` on invoice and back-link payment transaction `invoiceId` after materialization. |
</threat_model>
