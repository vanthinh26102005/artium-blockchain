# Phase 31 - UI Design Contract

> Visual and interaction contract for Orders invoice preview and extraction UI. Generated by gsd-ui-phase and verified before planning.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn-style shared UI with checked-in `components.json` |
| Preset | `radix-lyra`; Tailwind base color `neutral`; CSS variables enabled |
| Component library | Radix primitives through `@shared/components/ui`; use existing `Button`, `Dialog`, `Badge`, `Input`, and local Orders components before adding new primitives |
| Icon library | Use lucide-react for Phase 31 because Orders already uses lucide icons; do not introduce a second icon style inside Orders |
| Font | `ABC Monument Grotesk` app font for in-app surfaces; print document may fall back to system sans only inside print styles if browser print fidelity requires it |

Design sources:
- Phase 31 context decisions D-01 through D-20 are locked.
- Existing Orders workspace uses Tailwind utility classes, slate typography, white panels, `#F7F8FA` workspace background, and rounded panels.
- `FE/artium-web/components.json` confirms shadcn-style config is present; no shadcn initialization is needed.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, chip icon spacing, dense metadata separators |
| sm | 8px | Compact element spacing, status chip internal gaps |
| md | 16px | Default row gaps, line-item padding, mobile block spacing |
| lg | 24px | Panel padding, modal toolbar padding, document section padding |
| xl | 32px | Desktop document gutters, panel stack gaps, modal internal breathing room |
| 2xl | 48px | Major invoice document section breaks on desktop/print |
| 3xl | 64px | Top/bottom print document margins when viewport/page size permits |

Exceptions:
- Icon-only close buttons and compact modal controls must keep a minimum 40px by 40px hit target to match existing `DialogContent` close behavior.
- Print margins may use browser page margins, but all component-level padding must stay on the scale above.

Rules:

Accent must not be applied to every button, every link, or decorative invoice document elements.

Copy rules:
- Do not expose unauthorized/forbidden/not found distinctions from the invoice endpoint in the UI.

- `Download` may appear only as explanatory browser language in developer comments or docs, not as the UI command. The user-facing action is `Print invoice`.

## Screen And Interaction Contract

### Orders List

- Add a compact invoice chip to `OrderListCard` without turning the card into a dense action surface.
- The chip appears in both buyer and seller scopes when invoice preview or extraction can be attempted for the order state.
- The list chip navigates to `/orders/[orderId]?scope={scope}&invoice=1` or an equivalent query/focus mechanism that preserves scope and focuses the detail Invoice panel.
- The chip must not reset filters, pagination, role scope, or search state beyond normal navigation to detail.
- The list card remains the primary navigation target; invoice chip text and icon must be compact and status-aware.

### Order Detail

- Add a dedicated `Invoice` panel in the right-column stack, separate from `OrderActionPanel`.
- The panel must sit near `OrderActionPanel` and `Order totals` so lifecycle actions, totals, and invoice controls remain distinct.
- When opened from the list chip, focus should land on the Invoice panel heading or first enabled invoice action.
- The panel must show the current invoice availability state: `Checking`, `Ready`, `Unavailable`, or `Retry`.
- `Preview invoice` opens the modal. `Print invoice` calls the same print-ready document path and is disabled until invoice data is ready.

### Preview Modal

- Use the existing Radix/shadcn `Dialog` pattern.
- Desktop: large centered modal, max width near existing `8xl`/`9xl` sizes, with a scrollable document body and a fixed or visually persistent action row.
- Mobile: full-screen modal with the document first, compact top action row, and no horizontal scroll.
- Closing the modal returns to the same order detail context and preserves loaded order, action panel, timeline, payment records, and role scope state.
- The modal must handle loading independently if prefetch has not completed.

### Invoice Document

- The document should look like printable paper, not another dashboard card.
- Above the fold: invoice number, invoice status, order number, issue date, paid date when provided, and total.
- Party blocks: buyer and seller names/emails only when backend provides them; redacted or missing values are labeled.
- Line items: desktop and print use structured rows; mobile uses stacked blocks with title, description, quantity, unit price, tax/discount, and line total.
- Payment records: payment status, method, transaction ID, payment intent, wallet transaction hash, and on-chain order ID when provided.
- Totals: subtotal, tax, discount, shipping, and total are rendered from backend invoice DTO fields only.
- Include minimal Artium footer in print output.

## State Contract

| State | Trigger | UI Contract |
|-------|---------|-------------|
| Checking | Order loaded and invoice prefetch or modal load is in progress | Show compact skeleton/loader in chip, detail panel, and modal; disable print |
| Ready | `orderApis.getOrderInvoice(order.id)` returns invoice data | Enable preview and print; render document from returned DTO |
| Unavailable | Invoice endpoint returns non-disclosing authorization/not-found/unavailable response, or order state cannot expose invoice | Keep actions visible but disabled; show reason without leaking authorization detail |
| Retry | Network or retryable server failure | Show inline retry in panel/modal; chip says `Retry invoice`; keep order workspace usable |

Loading/error rules:
- Fetch invoice data lazily after order detail load, not before order detail is available.
- The modal must be able to request invoice data if the prefetch has not completed.
- Authorization failures render as `Unavailable`, not as a distinct forbidden state.
- Retry must not reload the whole order detail page.

## Responsive And Print Contract

| Surface | Contract |
|---------|----------|
| Desktop list | Invoice chip fits within existing card rhythm and does not push total/next-step content into awkward wrapping |
| Mobile list | Chip wraps below metadata or near next step without overlapping artwork, status badge, or total |
| Desktop detail | Right-column Invoice panel aligns with existing 340px minimum column and does not nest cards inside cards |
| Mobile detail | Invoice panel becomes full-width in the normal page stack |
| Desktop modal | Centered, large document preview with scroll containment and visible close/action controls |
| Mobile modal | Full-screen, no clipped text, no horizontal scrolling, controls remain reachable |
| Print | Hide app chrome, modal overlay/frame, controls, workspace background, filters, pagination, and navigation; print only invoice document plus minimal Artium footer |

Print rules:
- Use browser print/save-as-PDF through `window.print()`.
- Do not add a PDF rendering dependency in Phase 31.
- Print is disabled until invoice data is ready.
- Print output must preserve identifiers, totals, dates, spacing, and readable line items.

---

## Data Truth Contract

- The only invoice data source is `orderApis.getOrderInvoice`.
- Do not recompute subtotal, tax, discount, shipping, total, invoice number, or invoice status on the client.
- Client formatting may format numbers/dates for display, but must not change the financial source value.
- Do not invent placeholder parties, addresses, payment identifiers, or artwork values. Use `Not provided` or `Redacted by access rules`.
- Reuse or extend existing Orders presentation helpers for formatting, status labels, money, dates, and identifier trimming.
- Seller redaction from Phase 30 must remain visible as a labeled absence, not replaced with guessed buyer information.

---

## Accessibility Contract

- Modal must have a clear accessible title tied to the invoice number when available.
- Icon buttons require screen-reader text or visible text.
- Disabled invoice actions must expose the reason in adjacent text, not only through color.
- Keyboard focus must remain inside the modal while open and return to the triggering control on close when practical.
- Ready, unavailable, retry, and checking states must not rely on color alone.
