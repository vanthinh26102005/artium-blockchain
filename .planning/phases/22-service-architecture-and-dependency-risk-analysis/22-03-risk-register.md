# 22-03 Architecture Risk Register

## Scope

This ARCH-03 artifact turns the Phase 21 baseline plus Plan 22-01 role/communication analysis into current-state production concerns. It ranks risks and assigns downstream ownership, but it does not prescribe platform fixes, workload kinds, service exposure, or scaling settings.

## Risk scoring guide

| Level | Meaning |
|---|---|
| High | Can break core traffic, payment/order integrity, runtime startup, or operational safety if copied into production assumptions. |
| Medium | Can create reliability, maintainability, or security exposure without immediate total failure. |
| Low | Needs cleanup or confirmation, but impact is bounded or mostly documentation/config clarity. |

## Highest-priority production concerns

1. **Transport contract drift:** `notifications-service`, `events-service`, and `community-service` have mismatches between gateway assumptions and service bootstraps. These can break core internal traffic before business logic runs.
2. **Trust-boundary fragility:** Stripe raw-body verification, websocket handshake assumptions, Google OAuth token verification, GCS credential files, and Blockchain RPC credentials all cross sensitive boundaries and need explicit ownership.
3. **Singleton/background behavior:** `OutboxProcessor` and `blockchain-event-listener` are not request-only services. Their active-instance semantics must be understood before operations guidance is written.
4. **Persistence lifecycle ambiguity:** `DB_STRATEGY`, shared schema creation, isolated DB evidence, and `DB_SYNCHRONIZE` behavior need a controlled persistence model before release design.
5. **Non-authoritative artifact residue:** stale Dockerfile ports, missing `.env.compose` files, dev Kubernetes YAML, and `crm-service` evidence must not be promoted to production truth without revalidation.
