# 22-02 Communication Path Matrix

## Scope

This ARCH-02 artifact describes how traffic and events move through the current backend. It uses Phase 21 artifacts, `22-01-service-role-catalog.md`, and direct source evidence. It records communication shape and ambiguity only; it does not choose namespaces, service exposure, probes, autoscaling, managed replacements, or other platform design.

## Current communication ambiguities

| Ambiguity | Evidence | Why it matters |
|---|---|---|
| Messaging websocket duplication | Gateway has `/messaging` websocket provider; messaging-service also has its own `@WebSocketGateway({ cors: { origin: '*' }})` | Phase 23 needs a single ownership model after Phase 22 risk analysis; this phase records the duplication without resolving it. |
| notifications-service transport mismatch | Gateway registers `NOTIFICATIONS_SERVICE` as a TCP client, but notifications bootstrap starts HTTP only on `process.env.port ?? 3002` | Gateway notification routes may not have a live TCP target under current code. |
| events-service TCP-port ambiguity | Gateway config defaults events to `3107`, while events bootstrap falls back to `3109` | Non-Compose or incomplete env runs can misroute events RPC traffic. |
| community-service TCP-port ambiguity | Gateway config defaults community to `3109`, while community bootstrap falls back to `3106` | Non-Compose or incomplete env runs can misroute community RPC traffic. |
| Stripe webhook ownership | Phase 21 records both gateway and payments-service webhook/callback surfaces | Callback routing must preserve raw-body verification and one canonical owner later. |
| Missing `.env.compose` files | Compose references env files, while current checkout has no readable `BE/apps/**/.env.compose` files | Communication defaults may depend on untracked local files rather than repository truth. |
